# LexVoice

[English](README.md) | [简体中文](README.zh-CN.md)

LexVoice 是一个面向 Obsidian 桌面端的录音、转写、实时大纲和 AI 纪要沉淀插件。它把会议、访谈、课程、视频音频、个人口述和临时讨论整理成可阅读、可回听、可复用的 Markdown 笔记。

它不是云服务，也不自带 API Key。LexVoice 负责把录音、转写、AI 整理、沉淀入库和本地笔记写入串起来；真正的语音识别服务和大模型服务由你自己配置。

## 1.2.0 · 失踪人口回归，带回来一万行代码

这一版改得有点多——超过一万行代码动了。所以更像是一次重做，而不是更新。

LexVoice 一开始是个录音转写插件：录下来、转成文字、生成纪要，完事。但用了一段时间我发现，会议真正的价值不在那篇转写稿里——转写稿生成完那一刻，几乎就没人再打开了。值得留下来的是：谁说了什么、要做什么、学到了什么、下次还会用到的术语。这些才是会议结束后该带走的东西。

所以 1.2.0 把 LexVoice 从一个"转写工具"重做成了"会议工作台"——录音只是入口，后面的大纲、沉淀、对象库才是重点。顺便还做了亿点点美化工作，界面更优雅，交互更清晰。

### 主要新增

**实时大纲**

录音时章节边录边长。不用等录完再看到 AI 整理结果，会议进行到一半就能扫一眼"刚才聊了啥"。录完后章节和播放位置联动，点章节就跳到对应的录音位置。

**沉淀工作流**

每篇纪要扫一次，AI 把内容拆成四组候选：人员、待办、学习卡片、热词。每组用不同的决策模型——人员需要逐条裁决（留下/合并/忽略），其他三类默认全选、用户取消少数。处理完一组进下一组，跟流水线似的。

**对象库**

待办、学习卡片、人员、热词都能沉淀成 Obsidian 里的独立 Markdown 文件，进入你已有的知识库。不是导出，是变成可复用的对象——下次开会提到同一个人，能自动关联到已有档案。

**待办增强**

候选阶段就能编辑责任人、时间、子任务，全部行内编辑无弹窗。入库后写入标准 Markdown 待办格式，能被 Tasks 等插件识别。

**会中纪要 · 几个特殊符号**

录音过程中可以在侧边栏一边听一边记。普通文字就是普通备注，但如果首字符是这几个符号，会有不同的处理：

触发 AI 即时助理：

- `#概念` — 开会突然听到"既判力"不懂，打 `#既判力` 回车，AI 给出定义、用法、在当前讨论里的意义
- `?问题` — "刚才那个判例叫啥"，打 `?刚才那个判例` 回车，AI 结合当前转写和大纲直接回答
- `!重点` — "这条要单独跟进"，打 `!这是核心争议点` 回车，AI 标记为重点并说明最终纪要怎么处理

只标记不触发 AI：

- `@指派人` — 当场记下"这事 @腾哥 跟进"，最终纪要归并待办时会优先把任务挂到 ta 名下
- `/创建待办` — `/下周前出第二版方案`，最终纪要会把这条作为一个明确的待办归入待办候选

半角全角都认（`#` 和 `＃` 等价）。AI 不抢转写资源，会等到转写空档时再跑。所有会中备注会一起进入最终纪要的整理 prompt，作为"现场补充材料"标注清楚，不会被混进转写原文。

**录音侧的修复**

- 识别虚拟声卡输入，避免混合录制时麦克风被污染（之前会议被这个坑过几次）
- 录音前后都有电平条，能直接看到电脑音频和麦克风有没有真的有输入
- 删除转写记录时会问要不要一起删除录音文件

**导出**

一篇纪要可以继续生成 HTML 报告、HTML PPT、可编辑 PPTX、邮件草稿。共用同一份内容，按场景换皮。

**纪要列表**

侧边栏默认按本周和模板筛选，当前打开的纪要会高亮。以前在文件树里翻历史纪要很麻烦，现在直接在侧边栏时间轴看。

### 一堆细节修复

- 部分模板被错归类成"录音"导致筛选不准——修了
- 招聘评估、文本导入、整理失败这几个状态的展示都重新画了
- 沉淀扫描的进度、取消、重扫反馈做得更清楚
- 侧边栏右键菜单去掉了一堆不符合 Obsidian 习惯的项
- 边距、Tab 顺序、播放器、时间轴、候选密度全部重新调过
- 颜色系统改回完全跟随 Obsidian 主题变量（之前硬编码了一些粉色，跟 Style Settings 不兼容）

如果你之前用过 LexVoice 1.x，建议看看沉淀 tab 和会中纪要的几个符号——这是这一版最值得试的两个东西。

## 功能亮点

### 实时大纲

- 录音时同步生成大纲，适合边开会边确认当前内容结构。
- 大纲使用时间轴呈现，章节可点击跳转到对应录音位置。
- 录音中显示音频电平，帮助判断麦克风和电脑音频是否真的有输入。
- 录音结束后进入最终整理状态，完成后可继续沉淀人、事、知识点和热词。

### 沉淀工作流

- 新增 `大纲 / 沉淀 / 纪要` 三段式侧边栏工作流。
- 沉淀 Tab 一次扫描当前纪要，产出四组候选：人员、待办、学习、热词。
- 人员组支持逐条裁决：留下、合并、忽略。
- 待办、学习、热词支持多选加入：默认全选、全选、反选、忽略未选。
- 已处理的组可回看，也可重新处理本组。
- 入库后提供 Toast，可查看结果或撤销本次提交。

### 信息对象库

- 学习卡片：把概念、机制、案例、QA、追问和观点沉淀为可复用卡片。
- 待办卡片：把明确行动项沉淀为待办，并可同步写入日记中的 Markdown 待办。
- 人员资料：从纪要中识别候选人员，确认后进入人员库。
- ASR 热词：从当前纪要提取人名、品牌、机构、术语等，服务后续识别。
- 提供学习卡片瀑布墙、概念墙和待办墙入口。

### 真实麦克风保护

- 混合录制时区分电脑音频和真实麦克风。
- 避免把 `CABLE Output`、`BlackHole`、`VoiceMeeter`、`Stereo Mix` 等虚拟声卡输入误当成麦克风。
- 设置页提供真实麦克风选择和音频设备检测。
- 录音中电平条不动时，可以快速判断是麦克风没进来，还是电脑音频没进来。

### 导出与分享

- 可从纪要生成 HTML 报告，适合阅读、打印或分享。
- 可生成 HTML PPT，适合快速演示。
- 可生成可编辑 PPTX，方便后续在 PowerPoint 里继续修改。
- 可生成邮件草稿，自动附带 Markdown、PDF 和已生成的报告或 PPT 文件。

### 纪要导航

- 侧边栏纪要列表按日期时间轴展示。
- 默认显示本周，支持按模板筛选。
- 当前打开的纪要会在列表中高亮。
- 右键删除转写记录时，可选择是否同时删除对应录音文件。

## 适合谁

- 经常开会，希望自动生成纪要、决策、待办和后续动作的人。
- 做用户访谈、专家访谈、招聘面试或调研访谈的人。
- 看 B 站、YouTube、课程、讲座时，希望把音频内容整理成学习笔记的人。
- 用 Obsidian 管理知识库，希望把录音、纪要和知识对象都留在本地库里的人。
- 对隐私有要求，希望可以选择本地转写和本地大模型的人。

## 基本工作流

1. 在 LexVoice 侧边栏选择模板和音频输入。
2. 开始录音，确认麦克风/电脑音频电平有输入。
3. 录音时查看实时大纲，必要时点击章节回听或跳转。
4. 录音结束后等待 AI 整理最终纪要内容。
5. 进入沉淀 Tab，确认人员、待办、学习卡片和热词。
6. 将确认后的对象写入人员库、待办库、学习卡片库或热词库。
7. 需要分享时，生成 HTML 报告、HTML PPT、可编辑 PPTX 或邮件草稿。

默认路径：

- 录音文件：`LexVoice/录音`
- 转写纪要：`LexVoice/转写纪要`
- 学习卡片：`LexVoice/学习卡片`
- 待办卡片：`LexVoice/待办`
- 邮件草稿：`LexVoice/邮件草稿`
- 领域词汇表：`LexVoice/词汇表.md`

这些路径都可以在设置中自定义。

## 使用前需要准备什么

必需：

- Obsidian 桌面版
- 一个语音转文字服务：可以是云端 API，也可以是本地模型服务
- 一个用于保存录音的 Obsidian 库内文件夹
- 一个用于保存转写纪要的 Obsidian 库内文件夹

可选但推荐：

- 一个大模型 API：用于整理纪要、生成大纲、沉淀对象、生成报告和导出 PPT
- 一个虚拟音频设备：用于录制线上会议、视频播放或电脑声音
- 一个真实麦克风：混合录制时用来录下自己的声音
- 一个领域词汇表：用于提高人名、品牌名、项目名和行业术语的识别准确度

费用说明：

- Obsidian 和 LexVoice 插件本身不收取 API 费用。
- 云端语音转写和云端大模型通常由服务商按量计费，具体以服务商页面为准。
- 本地语音转写和本地大模型可以不产生 API 费用，但需要你自己安装服务，并承担电脑性能、显存和维护成本。

## 首次使用：先配置什么

![LexVoice 首次使用配置概览](docs/images/first-use-settings.svg)

第一次打开 LexVoice 设置页时，建议按下面顺序配置：

1. **配置语音转写服务**：这是录音能变成文字的前提。可以使用云端语音转写服务，也可以使用本地 Whisper、SenseVoice、faster-whisper 等服务。
2. **确认保存路径**：建议确认录音、转写纪要、学习卡片、待办卡片和词汇表路径。
3. **配置大模型 LLM API**：如果只需要原始转写，可以暂时不配置；如果需要 AI 整理、大纲、沉淀、导出和提示词优化，就需要配置。
4. **选择音频输入方式**：只录自己时选择仅麦克风；录课程或视频时选择仅电脑音频；线上会议或边听边讲解时选择麦克风加电脑音频。
5. **做一次设备检测**：确认电脑音频和真实麦克风都能被 LexVoice 识别。
6. **设置模板、日记、词汇表和提示词**：这些属于体验优化项，可以逐步补齐。

最小可用配置是：**语音转写服务 + 保存路径**。更完整的体验是：再配置 **LLM API + 音频设备检测 + 词汇表 + 自定义提示词**。

## 音频输入与真实麦克风

Obsidian 桌面端不能稳定、跨平台地直接捕获系统声音。如果你要录线上会议、B 站、YouTube、课程视频或电脑播放声音，通常需要安装虚拟音频设备。

常见工具：

- Windows：VB-Cable
- macOS：BlackHole
- Linux：PulseAudio 或 PipeWire 的 monitor source

Windows 使用 VB-Cable 时要注意命名反直觉：

- 把会议软件、B 站客户端、浏览器或系统声音的输出设备切到 `CABLE Input`。虽然名字叫 Input，但它在 Windows 里是“播放/输出设备”。
- LexVoice 读取的是对应的 `CABLE Output`，也就是这根虚拟线缆在“录制/输入设备”里的另一端。
- 如果你还要录自己的声音，LexVoice 的“真实麦克风选择”必须选择本机真实麦克风，不要选择 `CABLE Output`、`BlackHole`、`VoiceMeeter` 或 `Stereo Mix`。
- 录音中如果电平条不动，先打开“设备检测”确认当前输入是否真的有声音。

## AI 整理和提示词

AI 整理是可选功能。没有配置大模型时，LexVoice 仍然可以保存录音和转写文本；配置大模型后，可以获得更完整的整理能力。

内置模板围绕真实任务设计，例如工作纪要、学习笔记、访谈纪要、招聘评估和个人笔记。你也可以创建自定义模板，命名后会出现在录音、导入和重新整理流程中。

一个好的模板应该说明：

- 这段内容的使用场景
- 需要优先提取什么信息
- 是否需要待办、决策、风险、问题清单
- 是否需要翻译或双语输出
- 输出应该面向谁继续使用

## 文本和音频导入

- 可以导入已有音频，转写后整理为纪要。
- 可以导入已有文本或 Markdown，直接进入结构化整理。
- 重新整理已有纪要时，可以选择不同模板或补充偏好要求。
- 招聘评估等重型模板会把原始信息、匹配判断和风险点分开输出，方便后续复核。

## 导出能力

LexVoice 不只保存 Markdown，也可以把同一篇纪要转成不同交付物：

- **HTML 报告**：适合阅读、打印、分享和归档。
- **HTML PPT**：适合快速全屏演示或导出长图。
- **可编辑 PPTX**：适合在 PowerPoint 中继续排版和修改。
- **邮件草稿**：生成 `.eml` 文件，自动带上纪要摘要和相关附件。

相关文件夹可以在设置中配置。

## 诊断和排障

LexVoice 提供诊断报告，方便排查转写、队列、LLM、音频设备和配置问题。

诊断报告会自动隐藏常见 API Key、Token、用户目录和知识库路径；不会包含音频、转写正文或完整 Prompt。你可以在设置页复制诊断报告，再发给开发者定位问题。

## 安装方式

1. 关闭 Obsidian。
2. 将本仓库文件夹放到你的库目录：

   ```text
   <你的库>/.obsidian/plugins/lexvoice/
   ```

3. 重新打开 Obsidian。
4. 进入 设置 -> 第三方插件，启用 LexVoice。
5. 打开 LexVoice 设置页，先配置语音转写服务，再开始录音。

不要提交或公开本地的 `data.json`。它可能包含 API Key、服务地址和你的个人设置。

## 隐私和数据风险

LexVoice 没有自有云端存储，也不会把录音上传到 LexVoice 服务器。录音文件只保存在你选择的本地 Obsidian 库路径。

当你使用云端语音转写或云端大模型时，音频、转写文本、Prompt 和相关上下文会发送给你自己配置的服务商。请根据内容敏感度选择服务。

涉及涉密、隐私、客户资料、医疗、法务、人事、招聘、内部战略等内容时，建议优先使用本地语音转写和本地大模型，不要通过云端 API 处理。

录音前请遵守所在地法律法规，并在必要时征得参与者同意。

更多说明见 [PRIVACY.md](PRIVACY.md)。

## 已知限制

- 仅支持 Obsidian 桌面端。
- 暂不支持 Obsidian 移动端。

## 开发与构建

LexVoice 按 Obsidian 插件的常规工程结构维护：开发入口是 `src/main.ts`，执行 `npm install` 安装依赖，开发时运行 `npm run dev`，发布前运行 `npm run build` 生成 `main.js`。`main.js` 会继续提交到仓库并放入 Release，因为 Obsidian 实际加载的就是这个发布文件。
